# 读心助手 · 设计文档

> 一个能实时分析「对方心理状态」并给出应对建议的沟通助手。
> 应用场景：销售推进客户、与喜欢的人聊天/暧昧推进、以及通用的高情商沟通。

---

## 1. 产品目标

用户在和别人沟通时（微信文字、或当面/电话交谈），常常拿不准：

- 对方现在是什么心理状态？是有兴趣、在犹豫、还是已经反感？
- 我现在该推进（逼单 / 约见 / 表白 / 拉近关系）还是该慢下来？
- 具体该说什么？

本产品的目标：**随时给用户一个「场外军师」**，输入当前对话内容，立刻得到
①对方心理分析 ②是否适合推进的节奏判断 ③可直接发送/说出的话术。

设计底线：建议要**真诚、有分寸、尊重对方**，帮助用户做更有效的沟通，而不是
操纵、欺骗或施压。

---

## 2. 两种输入场景

产品的核心难点不是「分析」，而是「**怎么把对话内容拿进来**」。我们支持两条输入通道：

| 通道 | 场景 | 状态 |
| --- | --- | --- |
| **A. 聊天截图（OCR）** | 微信等文字聊天，用户截图上传 | ✅ 已实现 MVP |
| **B. 实时录音（ASR）** | 当面交谈 / 打电话，实时录音转写 | 🚧 设计中，骨架已搭 |

两条通道在「拿到一段带说话人区分的对话文本」之后，**汇合到同一个分析引擎**。
这是整个架构的关键：分析引擎与输入方式解耦。

```
            ┌─────────────┐
截图 ──OCR──▶│             │
            │  对话文本    │──▶ 分析引擎(Claude) ──▶ 结构化结果 ──▶ UI
录音 ──ASR──▶│ (谁说了什么) │
            └─────────────┘
```

## 3. 整体架构

```
┌────────────┐        ┌──────────────────────────────┐        ┌──────────┐
│  前端       │        │  后端 (Node + Express)        │        │  外部服务 │
│  浏览器     │        │                              │        │          │
│            │        │  /api/analyze   截图分析       │──────▶ │ Claude    │
│ - 截图上传  │──HTTP─▶│  /api/audio(ws) 实时音频流     │──────▶ │ API(视觉/ │
│ - 录音采集  │  WS    │                              │        │  文本)    │
│ - 结果展示  │◀──────▶│  lib/analyzer  分析引擎       │        │          │
│            │        │  lib/asr       语音转写适配器  │──────▶ │ ASR 服务  │
└────────────┘        └──────────────────────────────┘        └──────────┘
```

- **后端必须存在**：API Key 不能放进浏览器；ASR 的密钥同理。
- 分析逻辑集中在 `lib/analyzer`，两种输入通道复用同一套 prompt + 结构化 schema。
- ASR 通过 `lib/asr` 适配器接入，**可替换**（国内合规可换讯飞/阿里/腾讯，海外可用
  Whisper/Deepgram），上层不感知具体厂商。

### 当前代码结构

```

├─ server.js              # Express 入口；/api/analyze 已实现
├─ public/
│  └─ index.html          # 移动端单页：截图上传 + 结果展示（已实现）
├─ lib/

├─ DESIGN.md              # 本文档
└─ README.md             # 运行说明
```

## 4. 分析引擎（两通道共用）

输入：一段对话文本（或一张截图）+ 场景（销售/情感/通用）+ 可选背景。
输出：固定结构的 JSON（用 Claude 的 `output_config.format` 强约束），字段：

| 字段 | 含义 |
| --- | --- |
| `counterpart_state` | 对方此刻的心理状态与潜在想法 |
| `temperature` | 意向/好感热度 0–100 |
| `temperature_reason` | 给出该热度的依据 |
| `signals[]` | 关键信号（对方说了什么 → 含义） |
| `can_push` | 推进节奏判断：可以推进 / 保持节奏 / 需要缓一缓 |
| `push_reason` | 为什么，提醒分寸 |
| `replies[]` | 2–3 条可直接用的话术（话术 + 为什么这样说） |
| `avoid[]` | 此刻应避免的做法 |

> 截图通道用 Claude 多模态：直接把截图喂给模型，由它同时完成 OCR + 气泡归属
> （右=我，左=对方）+ 分析，一步到位，无需单独的 OCR 管线。

## 5. 实时录音通道（重点设计）

### 5.1 数据流

```
麦克风采集 ──▶ 分片音频(每 ~3-5s) ──WS──▶ 后端
                                          │
                                  ASR 流式/分片转写
                                          │
                              累积「对话稿」(带说话人/时间)
                                          │
                          触发器：每累积 N 句 或 每隔 T 秒
                                          │
                                  分析引擎(Claude) ──▶ 结果
                                          │
                                  ◀──WS── 推回前端，实时更新军师面板
```

### 5.2 几个硬问题与取舍

1. **ASR 选型（最关键，需用户拍板）**
   - 当面/电话是**中文连续语音**，需要流式或近实时转写。
   - 国内合规 + 中文效果：讯飞、阿里云、腾讯云实时语音识别。
   - 海外/统一栈：Whisper（自托管或 API）、Deepgram。
   - 适配器 `lib/asr.js` 抽象掉差异，先做一个「分片上传 → 整段转写」的简单实现
     跑通链路，再升级为真流式。

2. **说话人区分（diarization）**
   - 截图通道靠气泡左右天然区分；录音只有一路麦克风，区分「我 vs 对方」更难。
   - 方案分级：
     - **v1（最简单）**：不强行区分，把整段对话交给模型，靠上下文推断，
       UI 上让用户在开始时说明「我是男声/女声」等线索。
     - **v2**：用户在说话时按住「我在说」按钮做手动标注。
     - **v3**：接 ASR 厂商自带的 speaker diarization，自动分离。
   - 建议先做 v1 跑通，体验上够用。

3. **实时节奏 vs 成本**
   - 不必每句都调一次 Claude（贵且抖动）。采用「滚动分析」：每累积若干句或每
     隔 ~10–15 秒分析一次，只把**最近一段 + 简短历史摘要**送进模型。
   - 用 prompt caching 缓存系统提示与场景设定，降低重复成本。

4. **隐私与合规**
   - 实时录音涉及对方，需在产品上明确告知与授权边界。音频默认不落盘，转写文本
     仅用于当次分析。这点在正式上线前必须确认。

### 5.3 后端接口（计划）

- `GET /api/audio`（WebSocket）：
  - 客户端上行：`{type:"audio", chunk:<base64>}`、`{type:"config", scene, note}`、`{type:"end"}`
  - 服务端下行：`{type:"transcript", text, speaker?}`、`{type:"analysis", data}`、`{type:"error", msg}`

## 7. 待定决策（需用户确认）

1. **ASR 服务选哪家？** —— 决定 `lib/asr.js` 的具体实现与合规路径。
2. **说话人区分做到哪一级？** —— v1 不分 / v2 手动标注 / v3 自动 diarization。
3. **实时性要求？** —— 接近实时（每 10–15s 一次分析）是否够用，还是要逐句。
4. **平台优先级** —— 先继续做 Web，还是做手机原生 App（详见 §8，App 的价值几乎
   全在输入通道，且 iOS/安卓能力差别巨大）。

> 这些不影响现在继续搭框架；定了之后再落地 `lib/asr.js` 与 `/api/audio`。

## 8. App 形态设计

如果要做成手机 App，核心结论先说：**现有的后端与分析引擎（`lib/analyzer.js`、
结构化输出、场景 prompt）全部复用，App 只是换了个「壳」。** 架构如下：

```
iOS / 安卓 App(瘦客户端)  ──┐
                          ├─▶ 同一个后端 + 分析引擎 ─▶ Claude
现有 Web 页面           ──┘
```

### 8.1 App 的价值几乎全在「输入通道」

App 值不值得做，取决于它能否更顺地把对话内容拿进来。这里 **iOS 与安卓差别巨大，
是设计里最关键的现实约束**：

| 能力 | 安卓 | iOS |
| --- | --- | --- |
| 悬浮窗军师（浮在微信上实时显示建议） | ✅ `SYSTEM_ALERT_WINDOW` | ❌ 系统禁止悬浮在别的 App 上 |
| 后台/前台服务持续录音 | ✅ 前台服务 | ⚠️ 切到微信后基本录不了 |
| 截图「分享到」App | ✅ 分享菜单 | ✅ Share Extension |
| 读取微信聊天内容 | ❌（无障碍 hack，有风险） | ❌ |
| 一键截屏并分析 | ✅ | ⚠️ 需手动截图再分享 |

**结论：安卓能做出「悬浮军师 + 后台录音」的完整体验；iOS 基本只能做「截图分享 +
App 内当面录音」。** 这直接决定产品形态。

### 8.2 App 架构（以安卓完整形态为例）

```
┌─────────── App ───────────┐
│  输入层                    │
│   ├─ 截图分享入口(Share)    │
│   ├─ 录音服务(前台Service)  │──采集音频片段
│   └─ 悬浮窗(Overlay)       │──显示实时军师面板
│  ────────────────────────  │
│  通信层                    │──WebSocket/HTTP──▶ 后端(复用) ─▶ Claude
│  ────────────────────────  │
│  本地层                    │
│   └─ 历史会话 / 隐私开关(本地) │
└───────────────────────────┘
```

三条输入通道映射到 App：
1. **截图** → 系统分享菜单丢进 App → 走现有 `/api/analyze`
2. **当面录音** → App 内录音（两端都行）→ WebSocket → ASR → 滚动分析
3. **安卓专属** → 悬浮窗 + 后台录音，边聊边在屏幕角落弹建议

### 8.3 技术选型

- **跨平台框架**（React Native / Flutter）能共用业务与 UI，但**悬浮窗、前台录音
  服务必须写原生模块**，跨平台省不掉这部分。
- 建议：若核心卖点是「实时军师悬浮窗」，**安卓优先**（原生 Kotlin，或 RN + 原生
  模块）；iOS 作为「截图分析 + 当面录音」轻量版跟进。
- 后端与 Web 版完全不动。

### 8.4 隐私与合规（App 尤其重要）

录音涉及对方、悬浮窗读屏都是敏感能力，上架审核（iOS App Store、国内应用商店）会
严格审查。必须有明确的录音授权提示、音频不落盘承诺、悬浮窗权限引导。这会直接影响
能否上架，是产品能否成立的前提，不是细节。

### 8.5 App 相关的待定决策

1. **平台优先级**：安卓优先（完整体验，需原生模块）／ iOS 优先（轻量版）／
   两端跨平台同时做 ／ 先不做 App、继续验证 Web。若目标是 Windows + iPhone
   组合，见 §9。
2. **跨平台框架**：React Native、Flutter，还是各写原生。取决于团队技术栈与对
   悬浮窗/录音等原生能力的依赖程度。
3. **合规路径**：录音授权与上架策略，尤其国内商店对「录音 + 读屏」类应用的要求。

## 9. Windows 桌面端 + iPhone 双端方案

如果目标是**同时支持一个 Windows 电脑应用 + 一个 iPhone App**，核心结论：

1. **后端与分析引擎依然一行不改**，两端都是瘦客户端，架构与 §8 相同。
2. **Windows 端是所有平台里能力最强的**——桌面系统没有 iOS/安卓的沙箱限制，
   而且**微信电脑版就在同一块屏幕上**，「实时军师」体验在 Windows 上反而最完整。
3. 两端能力差距太大，**不建议强行一套代码**；按「各取所长 + 共用后端」来做。

### 9.1 能力对比：Windows vs iPhone

| 能力 | Windows | iPhone |
| --- | --- | --- |
| 悬浮军师面板（置顶小窗，贴在微信旁） | ✅ 置顶窗口随便开 | ❌ 不能悬浮在别的 App 上 |
| 自动捕获微信聊天窗口（截屏→分析，无需手动截图） | ✅ 窗口捕获 API / 热键触发 | ❌ 只能手动截图再分享 |
| 持续录音（当面 / 电话 / 会议） | ✅ 麦克风 + 可采集系统声音（网课/语音通话） | ⚠️ 仅 App 在前台时 |
| 全局热键（一键分析当前微信窗口） | ✅ | ❌ |
| 便携性（出门当面聊） | ❌ 坐在电脑前才有用 | ✅ 核心价值 |

### 9.2 Windows 端设计（能力主场）

**杀手级形态：置顶军师面板 + 一键捕获微信窗口。**

```
┌── 用户屏幕 ──────────────────────────────┐
│  ┌─ 微信电脑版 ─────┐   ┌─ 读心助手(置顶) ─┐ │
│  │  正在和客户聊…    │   │ 热度 62 ↗        │ │
│  │                 │   │ 建议：可以推进…    │ │
│  └─────────────────┘   └─────────────────┘ │
│        ▲ 热键/定时 → 捕获微信窗口 → /api/analyze │
└─────────────────────────────────────────┘
```

- **输入通道 1：窗口捕获**。按全局热键（或每隔 N 秒自动）对微信窗口截屏，
  直接走现有 `/api/analyze`——比手机截图分享还顺，聊天过程中零打断。
- **输入通道 2：桌面录音**。麦克风（当面/电话外放）+ 系统声音回采（微信语音
  通话、线上会议），走 §5 的 WebSocket 音频通道。
- **技术选型**：**Tauri 或 Electron 包一层现有 Web 前端**（`public/index.html`
  几乎直接复用），窗口捕获/全局热键/置顶用框架原生 API 即可，无需写多少原生代码。
  这是三种客户端里开发成本最低的一个。

### 9.3 iPhone 端设计（便携主场）

受 §8.1 所述 iOS 限制，iPhone 端做**轻量版**，聚焦两件事：

1. **截图分享**：微信截图 → 系统分享 → Share Extension → `/api/analyze`。
2. **当面录音**：出门见客户/约会时，App 前台录音 → 音频通道 → 实时军师。

技术选型：原生 Swift（体量小）或 React Native。不必等 Windows 端一起发版。

### 9.4 复用策略：共享什么、不共享什么

```
        ┌────────── 共享（改一处，三端生效）──────────┐
        │  后端 API + lib/analyzer + prompt/schema   │
        └───────────────────┬───────────────────────┘
     ┌──────────────┬───────┴────────┬──────────────┐
   Web(现有)   Windows(Tauri,        iPhone(Swift/RN,
              复用 Web 前端+捕获/录音)  分享扩展+录音)
```

- **共享**：全部分析逻辑、API 协议、结果数据结构；Windows 端还能直接共享 Web UI。
- **不共享**：各端输入通道（窗口捕获 / Share Extension / 录音采集）——这正是
  每个端的价值所在，本来就没法共享。
- **不建议** Flutter/RN 一套代码打 Windows + iOS：两端核心能力都在原生层，
  UI 层省下的钱抵不上跨平台桌面端的坑。

### 9.5 落地路径（按此顺序执行）

**第 1 步：Windows 端骨架（成本最低、收益最大，约 1–2 周出雏形）**

用 Tauri（备选 Electron）把现有 `public/index.html` 包成桌面应用，再加三个原生能力：

1. **窗口置顶**——军师面板贴在微信旁边，一个窗口配置项。
2. **全局热键**（如 `Ctrl+Shift+A`）——按下时对微信窗口截屏，直接走现有
   `/api/analyze`，聊天零打断。
3. **窗口捕获**——Windows Graphics Capture API，Tauri/Electron 均有现成插件。

这一步几乎全是「组装」，现有截图分析链路原样复用，**不依赖任何待定决策**，
可以立刻开工。

**第 2 步：录音通道落地（= §6 的 M3，三端共用）**

Windows 端做录音比手机容易：麦克风 + 系统声音回采，可覆盖微信语音通话/线上会议。
**前置条件：先拍板 §7 的 ASR 选型**（讯飞/阿里 vs Whisper），随后实现
`lib/asr.js` 与 `/api/audio`（WebSocket）。

**第 3 步：iPhone 轻量版**

Swift 或 React Native，只做两件事：截图分享（Share Extension → `/api/analyze`）
+ 当面录音（走第 2 步已通的音频通道）。等音频通道跑通后再启动，不与
Windows 端绑定发版。

两端场景天然互补：**电脑前 → Windows 管，出门 → iPhone 管**，不互相抢活。
